- **Lead takeover rate was 71% (5/7 phases).** Symptom: only `tester-review` orchestrator and `pm-doc-writer` ran as actual subagents in R3; PM Manager, Dev, Tester Diff, Tester Playwright, and (effectively) PM Triage were all lead-written. Root cause: the PM was thinking "users = humans reading the UI" but the change is for "users = agents reading the JSON payload". ## Skill gaps found (changes that would have prevented the issue)

### Gap 1: Worktree path is hardcoded, not portable

- **Symptom**: `SKILL.md:455` hardcodes `/Users/yangweibin/.worktrees/team-dev-loop-round-N`. R3's plan.md (`.omo/round-3/plan.md:50`) inherited this. On Linux env, the path doesn't resolve. R3 dev phase committed directly to main, contradicting the worktree mandate. R1+R2 used remote branches (`origin/team-dev-loop-round-1-...`, `origin/team-dev-loop-round-2-...`) which is a different convention from what SKILL.md says.
- **Existing-skill-text that didn't catch it**: `SKILL.md:455` — single hardcoded macOS path; no fallback for Linux/Windows; no rule about when to use worktree vs commit-direct-to-main.
- **Proposed patch**: (a) Replace the hardcoded path with a template: `WORKTREE_DIR="${WORKTREE_DIR:-$HOME/.worktrees/team-dev-loop-round-$N}"` with `WORKTREE_DIR` env override; (b) add a "Commit strategy" section that explicitly says: small bugfix → commit direct to main, feature/architecture → use worktree; (c) update `plan.md` template (in `references/phase-prompts.md`) to also use the templated path.

### Gap 2: Multi-round AC test design not documented

- **Symptom**: AC6 ("round 1 returns 3 findings, user resolves 1, round 2's payload includes that finding in `resolved[]`") cannot be verified by the existing e2e harness because the harness runs one session = one state.json = one server boot. R3's initial e2e scenario `resolved-context-payload` only verified state.json shape, not actual behavior, which caused Lens Goal FAIL.
- **Existing-skill-text that didn't catch it**: `references/phase-prompts.md` (Architect prompt) has a "Test plan" template that doesn't call out this harness limitation; the PM Manager's "SPECULATION" check doesn't catch "AC that can't be tested by available infrastructure".
- **Proposed patch**: (a) Add a "Harness limitation" section to `SKILL.md` (or `references/loop-decision.md`) that says: "if an AC requires multi-round behavior (round N depends on round N-1 state), the e2e harness cannot verify it — design the AC test as a unit test that directly invokes the function with a synthetic `Done` containing the prior-round state"; (b) add a checklist item to the Architect's plan template: "for each AC, identify whether it needs multi-round evidence; if yes, design a unit test, not an e2e scenario"; (c) update the Lens Goal prompt to check this.

### Gap 3: Lead-takeover rate is high; some phases should default to lead-written

- **Symptom**: R3 had 5/7 lead takeovers (PM Manager, Dev, Tester Diff, Tester Playwright, PM Triage-via-reframe). R1 had 3/7. The skill currently treats lead takeovers as a "design feature" but doesn't say which phases are "typically lead-written" by default.
- **Existing-skill-text that didn't catch it**: `SKILL.md` "Lead inline takeover protocol" section treats all takeovers as exceptional; no phases are marked as "typically lead-written".
- **Proposed patch**: Add a column to the per-phase execution table (around `SKILL.md:155-175`) called "Default executor" with values: PM Triage (subagent), PM Manager (subagent), Architect (subagent), Dev (subagent), Tester Review orchestrator (subagent), Tester Diff (lead by default — it's just `git diff main`), Tester Playwright (lead by default for small changes; subagent for UI-heavy changes), PM Doc Writer (subagent), Decision (lead always), Retro (lead always). This makes the takeover rate transparent and saves the orchestrator from spawning subagents that will return empty.

### Gap 4: Backlog-freshness gate — agent should surface fresh user-stories, not just pick from backlog

- **Symptom**: R1-R3 backlog (`follow_up_candidates` in `.omo/proposals.jsonl`) is dominated by small bugfixes (#3 Reopen end_line, #4 E2E coverage gap, #5 take-screenshots.mjs dead code). R3 PM Triage correctly surfaced a fresh user-story (prior-round context in payload) by self-investigation, not from backlog. The "fresh user-story" pattern worked but is implicit, not explicit in the skill.
- **Existing-skill-text that didn't catch it**: `references/phase-prompts.md` PM Triage prompt allows self-investigation but doesn't require it. The "Stop conditions" in `loop-decision.md` don't differentiate "backlog empty + agent has nothing" from "backlog has stale items + user wants fresh thinking".
- **Proposed patch**: (a) Add a "Backlog freshness gate" to the PM Triage prompt: if all backlog items are small bugfixes, PM MUST also do self-investigation (read README + recent code + recent commits) to surface at least 1 fresh user-story; (b) add this rule to the SKILL "Per-phase execution" table as a Phase 0 requirement when `follow_up_candidates` is bugfix-heavy.
