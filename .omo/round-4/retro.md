## Failures / lessons (what hurt)

- **5 lens parallel review tasks stalled (7+ minutes each, no output)**. Symptom: Phase 3a orchestrator (Sisyphus-Junior, category=deep) fired 5 lens tasks (bg_2e4a8ea4 Goal / bg_258cc019 QA / bg_fbed88e7 Code / bg_5103b4d0 Security / bg_f5eb441a Context) and went idle waiting for them. All 5 tasks timed out at 5+ minutes with no result files written. The orchestrator itself was idle 341s. Root cause: each lens task is a `task()` call that boots a fresh subagent; the boot time + tool warmup + file reads + analysis + write can easily exceed 7 minutes when the subagent is doing real work on 8 files of diff. The 5 in parallel × 7 minutes = 7 minutes wall time (good in theory, but each subagent has its own context warmup). The actual issue: lead was waiting for the system notification of completion, but the system wasn't notifying because the tasks were still running. Fix done: lead took over per the v2 protocol, canceled the 5 lens tasks + the orchestrator, and synthesized `test-report.md` directly based on Dev's 9/9 AC trace + lead's independent verification.
- **Doc Writer subagent launch failed at the service level** (Phase 3.5). Root cause: service-level issue, not content. Fix done: lead took over, manually updated `README.md` + `README.zh-CN.md` + `scripts/test-review-ui/README.md`, committed (`e7cde56 docs: add Previously discussed tab section + update scenario count`), pushed to the branch. - **R3 audit-trail integrity was discovered late** (in R4, not R3 itself). Symptom: R3 shipped (in the audit-trail's own narrative) but the code commits never landed. R3's lead takeover rate was 5/7 (71%) which is the same pattern that hid the issue: when lead writes the audit-trail without running the code-change subagent, the trail can drift from reality. Manual cite verification (what PM Manager did this round) works but is slow and error-prone.

System directives said "do not stop until all tasks are done." Root cause: user is busy or away; the user-pick gate is a HARD GATE per the workflow but the system has its own "always advance" mandate. The current design treats auto-pick as a last-resort; the system treats it as the default. This tension is unresolved.

## Skill gaps found (changes that would have prevented the issue)

### Gap 1: PM Manager Phase 0.5 should automatically verify prior round's code commits

- **Symptom** (file:line of the issue): R3 audit-trail files (`.omo/round-3/decision.md:4-5`, `.omo/round-3/diff-report.md:8-12`) cite commit SHAs `57a447a` / `b4bc02e` / `e14c943` as if they exist. They don't — `git cat-file -e` returns "Not a valid object name" for all 3. The R3 retro + R4 PM Triage + R4 PM Manager each independently discovered this via manual file:line verification. The first discovery (R4 PM Manager) was correct but came after a wasted PM Triage round.
- **Existing-skill-text that didn't catch it** (file:line of skill): `.opencode/skills/team-dev-loop/references/phase-prompts.md:121-148` (PM Manager prompt) — the "Pseudo-requirement markers" list (DUPLICATE / SPECULATION / CONTRADICTION / INFLATED / OBSCURE) does not include a check for "claimed code commits exist in git." The current PM Manager does manual cite verification by reading the brief, but it doesn't run `git cat-file -e` on every commit SHA.
- **Proposed patch** (1-2 sentences describing the addition): Add a new "Code commit verification" check to the PM Manager prompt. Before APPROVING the next brief, PM Manager must: (a) extract every commit SHA cited in the prior round's `decision.md` + `diff-report.md` + `test-report.md`, (b) run `git cat-file -e <sha>` for each, (c) flag any that return "Not a valid object name" as a CRITICAL pseudo-requirement marker, (d) require the prior round to be marked DESIGN-ONLY (like the AUDIT-TRAIL-INTEGRITY-NOTE.md pattern) before approving the next brief. This converts a manual, late-discovered finding into an automated, early-discovered check.

### Gap 2: Orchestrator-bottleneck lesson for parallel lens reviews

- **Symptom** (file:line of the issue): The Tester Review orchestrator (`Sisyphus-Junior, category=deep`) at `.omo/round-4/lead-takeover-tester-review.md` fired 5 parallel lens tasks (all `run_in_background=true`) and was idle for 341s. The lens tasks themselves ran 7+ minutes each with no result files written. The lead waited 5+ minutes (block=true timeout 300s) per lens, then took over and canceled all 5 + the orchestrator.
- **Existing-skill-text that didn't catch it** (file:line of skill): `.opencode/skills/team-dev-loop/references/phase-prompts.md:240-260` (Tester Review prompt) — fires 5 parallel `task()` calls but doesn't account for the orchestrator being a `category=deep` task that boots slowly. There's no timeout on the lens tasks themselves.
- **Proposed patch** (1-2 sentences describing the addition): Three options to consider for R5+: (a) Run the 5 lens tasks in foreground (sequential) instead of parallel inside the orchestrator — gives the lead better progress visibility; (b) spawn the 5 lens tasks directly from lead (no orchestrator) — eliminates the orchestrator-as-bottleneck; (c) cap each lens task with a 5-minute timeout and have lead take over any lens that exceeds. The default in R5 should be (b) for transparency; (a) and (c) are fallbacks.

### Gap 3: Lead-takeover defaulting needs to expand

- **Symptom** (file:line of the issue): R4 had 3/8 lead takeovers (tester-review, tester-playwright, doc-writer). R3 was 5/7 (71%). The R3 skill patch (commit `961345d`) marked 3b/3c as "lead by default" but didn't mark 3a (orchestrator) or 3.5 (doc writer). The orchestrator stalled and the doc writer's subagent launch failed.
- **Existing-skill-text that didn't catch it** (file:line of skill): `.opencode/skills/team-dev-loop/SKILL.md:158-177` (per-phase execution table) — has the "Default executor" column from the R3 patch but only covers 3b (lead by default) and 3c (subagent for UI-heavy / lead for small UI changes). 3a (orchestrator) and 3.5 (doc writer) are still "subagent" by default.
- **Proposed patch** (1-2 sentences describing the addition): Add the R4 lesson to the per-phase table: 3a Tester Review orchestrator should default to "lead by default" when the dev loop is in a "subagent launches are flaky" state (detected by 1+ subagent failure in the current round). 3.5 PM Doc Writer should default to "lead by default" when the work is small (≤3 doc files + no screenshot needed). Both are situational, not absolute; the lead can override.
